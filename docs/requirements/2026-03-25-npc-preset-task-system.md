# NPC预设任务系统需求冻结

日期：2026-03-25

## 目标

在现有 NPC 同伴系统之上，新增一层由模组驱动的“预设任务”能力。该能力用于让后端稳定执行模组作者预先定义好的 NPC 行为链，而不是让 LLM 每次临时自由编排步骤。

本轮需求重点服务《艾米莉》相关的两条任务线：

1. 独立调查：艾米莉离场 8 回合完成调查，返回次卧并向玩家交付调查报告，解锁新的逃脱路线。
2. 配合逃脱：艾米莉先吸引管家，玩家可选择接手吸引或背叛艾米莉，任务结果会影响信任与后续合作。

## 当前基础

代码中已经具备以下可复用基础：

1. `trust_level` 作为 NPC 硬状态。
2. `companion_mode` / `companion_task` 作为同伴任务执行骨架。
3. `follow` / `wait` / `bait` 三种硬行为。
4. 管家追逐状态、门阻隔、首轮追逐宽限、地图位置同步。
5. 隐藏地点的 `reveal_conditions`，可由线索、物品、flag 解锁。

因此本轮不重做 NPC 状态机，只在其上增加一层更薄的任务编排。

## 需求范围

### 1. 模组可声明预设任务

模组新增 `preset_tasks` 配置。每个任务至少包含：

1. `task_id`
2. `actor`
3. `requirements`
4. `steps` 或等价的固定阶段定义
5. `success_effects`
6. `failure_effects`

第一版只为当前模组支持有限动作，不设计通用 DSL。

### 2. 后端可读取并启动预设任务

后端应支持：

1. 按 `task_id` 启动任务。
2. 将任务状态写入 NPC runtime。
3. 在每回合推进中自动推进任务。
4. 任务完成或失败时写回持久状态。

第一版新增的 runtime 字段可采用：

1. `preset_task`
2. `preset_task_history`

### 3. 独立调查任务：`solo_search_escape`

艾米莉满足任务触发条件后，可执行独立调查任务。该任务的最终效果以“调查报告”而不是“直接瞬间结局”呈现。

#### 触发要求

1. 艾米莉信任值达到允许合作的阈值。
2. 玩家主动提出由自己吸引管家、由艾米莉独立调查全屋。
3. 任务必须通过后端命中模组中的 `task_id`，不能让 LLM 自由拼装步骤。

#### 执行效果

1. 任务启动后，艾米莉从地图与场景中消失。
2. 艾米莉离场持续 8 回合。
3. 8 回合结束后，艾米莉回到 `guest_bedroom`。
4. 完成时只写入 `emily_report_pending=true`，不立即将报告塞给玩家。
5. 玩家下次与艾米莉发生对话时：
   - 艾米莉输出模组内写死的调查报告内容
   - 玩家获得线索 `艾米莉的调查报告`
   - 写入 `emily_report_delivered=true`
   - 清除 `emily_report_pending`

#### 地图与结局效果

1. 新增隐藏地点 `emily_exit`。
2. `emily_exit` 的系统内部名称保持唯一，不与现有 `black_pit` 冲突。
3. `emily_exit` 的玩家可见名显示为 `纯黑地洞`。
4. `emily_exit` 的显性条件为玩家持有线索 `艾米莉的调查报告`。
5. 玩家进入 `emily_exit` 时触发新结局 `emily_escaped`。

### 4. 配合逃脱任务：`cooperative_escape`

该任务用于脚本化“艾米莉先引走管家，玩家决定是否接手”的合作战术。

#### 启动阶段

1. 艾米莉先移动到 `second_floor_hallway`。
2. 管家追逐目标切换为艾米莉。
3. 玩家获得可移动空间。

#### 分支一：玩家接手

1. 玩家进入 `second_floor_hallway`。
2. 首次激活接手的轮次不进行闪避检定，体现配合吸引。
3. 管家追逐目标切换为玩家。
4. 艾米莉切回 `follow player` 或等价可同行状态。

#### 分支二：玩家背叛

1. 玩家不接手，而是直接移动离开。
2. 艾米莉无法脱离二楼走廊，任务失败。
3. 艾米莉退回 `guest_bedroom`。
4. 艾米莉信任值降为 0。
5. 追加拒绝对话或拒绝合作的硬状态/flag。

### 5. LLM 的职责边界

LLM 在第一版中只允许：

1. 识别玩家是否在请求某个预设任务。
2. 输出结构化 `preset_task_request`，其中只包含：
   - `target_npc`
   - `task_id`
   - 可选 `reason`

LLM 不允许：

1. 自由写出任务步骤。
2. 直接改动 `preset_task` runtime。
3. 自由发放调查报告或结局线索。

### 6. 可见性与沉浸感

为了增强沉浸感：

1. 调查报告文本由模组提供，不由 LLM 即兴虚构。
2. 只有在艾米莉返回并与玩家完成一次对话后，报告才真正交付。
3. 线索的获得既是系统条件，也应成为可被 NarrativeAI 使用的叙事事件。

## 验收标准

### A. 独立调查链

1. 玩家提出“我吸引管家，你去查全屋”时，系统可命中 `solo_search_escape`。
2. 艾米莉离场后 8 回合内不在地图与场景中出现。
3. 第 8 回合后艾米莉出现在次卧。
4. 艾米莉不会在未对话前直接把线索塞进玩家状态。
5. 玩家和艾米莉对话后获得 `艾米莉的调查报告`。
6. 新隐藏地点 `emily_exit` 变为可见。
7. 玩家进入 `emily_exit` 后触发 `emily_escaped`。

### B. 配合逃脱链

1. 玩家可触发 `cooperative_escape`。
2. 艾米莉先到二楼走廊并吸引管家。
3. 玩家接手时能完成追逐目标切换。
4. 玩家背叛时艾米莉会退回次卧，信任变为 0，并拒绝合作。

## 非目标

本轮明确不做：

1. 通用行为树或通用脚本语言。
2. 多 NPC 并行协同任务系统。
3. 任意 NPC 的自动任务规划。
4. 完整战术 AI。
5. 对所有结局系统进行重构。

## 设计约束

1. 以最少代码改动复用现有 `companion_task`、回合推进、地图显隐和结局机制。
2. 不引入与现有地点 `name -> key` 解析相冲突的同名内部地点。
3. 玩家可见名允许与旧地点一致，但系统内部 key 和真实名称必须唯一。
